# Token Usage & Cost Calculation

## Overview

RunClaude reads Claude Code's JSONL log files and extracts token usage data in real time. This document explains how tokens are counted, how costs are estimated, and how data is aggregated at multiple time scales.

---

## 1. Data Source

Claude Code writes one JSONL file per workspace session to:

```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

Each line is a JSON event. `JSONLParser` scans each line for an `assistant`-type entry that contains a `usage` block.

---

## 2. Token Counting

Four token types are extracted from each `usage` block:

| Field | Description |
|---|---|
| `input_tokens` | Tokens sent in the prompt (not cached) |
| `output_tokens` | Tokens generated by Claude |
| `cache_creation_input_tokens` | Tokens written to the prompt cache for the first time |
| `cache_read_input_tokens` | Tokens read back from an existing cache entry (cheap) |

```
totalTokens = input + output + cacheCreation + cacheRead
```

Each record also carries the model name (e.g. `claude-sonnet-4-6`) and a timestamp, which are used for cost calculation and aggregation.

### Why totals differ from Claude Code's `/stats`

In a typical Claude Code session, **cache read tokens dominate the count by a large margin**. The system prompt and full conversation history are cached after the first turn, so every subsequent turn re-reads the entire context cheaply.

Real example from a single session (97 assistant records):

| Token type | Count | Share |
|---|---|---|
| `input_tokens` | 178 | < 0.01% |
| `output_tokens` | 59,756 | 0.9% |
| `cache_creation_input_tokens` | 610,476 | 9.3% |
| `cache_read_input_tokens` | 5,887,967 | **89.8%** |
| **Total (all 4)** | **6,558,377** | |
| input + output only | 59,934 | |

RunClaude currently includes all four types in `totalTokens`. Claude Code's `/stats` likely reports `input + output` only (or a different subset), which can produce a **~100Ă— difference** in the displayed total.

The **cost calculation is unaffected** by this â€” each type is already priced at its own per-million rate, so the cost figure remains accurate regardless of which tokens are included in the "total" display.

## 3. Cost Calculation

Costs are estimated using hardcoded per-million-token rates from `CostCalculator.swift`. The model is matched by name prefix.

### Pricing table (as of early 2026)

| Model | Input | Output | Cache write | Cache read |
|---|---|---|---|---|
| Claude Opus 4 | $15.00 | $75.00 | $18.75 | $1.50 |
| Claude Sonnet 4 / 3.5 | $3.00 | $15.00 | $3.75 | $0.30 |
| Claude Haiku | $0.80 | $4.00 | $1.00 | $0.08 |

Unknown models fall back to Sonnet-tier pricing.

### Formula

```
cost = (inputTokens   Ă— inputRate
      + outputTokens  Ă— outputRate
      + cacheCreation Ă— cacheCreationRate
      + cacheRead     Ă— cacheReadRate) / 1_000_000
```

Each token type is billed at its own rate because cache reads are significantly cheaper than fresh input tokens.

## 4. Aggregation Layers

`TokenAggregator` maintains three simultaneous layers of aggregation:

### 4.1 Sliding window (45 seconds)

Used to compute **tokens per second**, which drives the menu bar animation speed. Samples older than 45 s are pruned on every ingest. The 45 s window keeps the animation running through natural gaps in Claude Code output (tool calls, thinking, file reads).

```
tokensPerSecond = windowTokens / effectiveWindowDuration
```

### 4.2 Daily totals

Accumulates input, output, cache creation, and cache read tokens separately for the current calendar day. Resets at midnight. Broken down by model for per-model cost display.

### 4.3 Per-file sessions

One `FileSession` entry is maintained per JSONL file path (= one Claude Code workspace). Tracks:

- `firstRecord` / `lastRecord` timestamps
- `totalTokens`
- `modelBreakdown` â€” per-model token counts for accurate cost calculation

**Session cost** = sum of `CostCalculator.cost(for:)` across each model's usage breakdown.

**Burn rate** (tokens/min):
```
burnRate = totalTokens / elapsedMinutes
```

**Projected cost** (linear extrapolation to a full 8-hour session):
```
projectedCost = sessionCost Ă— (480 / elapsedMinutes)
```

A session is considered **active** if its last token record arrived within the past 30 seconds. Sessions disappear from the Live tab after 5 minutes of inactivity.
